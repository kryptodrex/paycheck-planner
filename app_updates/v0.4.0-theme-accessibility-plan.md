# v0.4.0 Theme and Accessibility Plan

Status keys:
- `[ ]` Planned
- `[-]` In Progress
- `[x]` Done

Release notes discipline:
- [ ] Add a user-facing RELEASE_NOTES.md bullet when each parent item reaches Done.
- [ ] Keep release-note wording focused on outcomes, not implementation details.

---

## 1. Appearance Architecture Foundation
Parent status: `[ ] Planned`

- [ ] Extend app-level settings to support appearance and accessibility preferences.
    - Keep these preferences in app settings, not plan files.
    - Include a clear model for theme mode, preset selection, custom appearance seed values, high contrast, and font scaling.
- [ ] Refactor theme application so the app resolves appearance from a single source of truth.
    - Avoid scattered `localStorage` theme reads outside the shared settings/theme flow.
- [ ] Define the semantic token contract for themeable UI.
    - Background, surface, text, border, accent, status, focus, overlay, and elevation tokens should be centrally owned.
- [ ] Audit existing raw or one-off styling patterns that would block stable theming.
    - Prefer semantic CSS variables in shared/global theme layers rather than component-local raw colors.
- [ ] Add/update tests for settings persistence and theme resolution behavior.

Done definition:
- [ ] Theme and accessibility preferences are persisted safely, applied consistently, and resolved from one predictable settings model.

---

## 2. Curated Theme Presets
Parent status: `[ ] Planned`

- [ ] Add curated preset theme families with paired light and dark variants.
    - Start with a small, opinionated set that feels polished rather than exhaustive.
- [ ] Separate theme mode from theme family.
    - Users choose Light, Dark, or System independently from the selected preset family.
- [ ] Ensure all major surfaces and shared controls render correctly across presets.
    - Include buttons, tabs, cards, modals, alerts, form fields, badges, and focus states.
- [ ] Add a lightweight preview pattern in Settings so users can understand the preset before leaving the modal.
- [ ] Add/update tests where necessary.

Done definition:
- [ ] Users can switch among a curated set of polished preset themes and get consistent results in both light and dark modes.

---

## 3. Opinionated Custom Theme Controls
Parent status: `[ ] Planned`

- [ ] Add a simple custom appearance mode driven by a few key color choices.
    - Keep the control surface intentionally small to avoid confusing users.
- [ ] Decide the initial editable inputs.
    - Recommended: primary accent and one or two supporting surface/tint controls only.
- [ ] Generate the rest of the semantic theme tokens from those core inputs.
    - Do not expose every CSS variable directly.
- [ ] Auto-generate a dark variant from the chosen custom colors.
    - Store explicit generated values so behavior is stable and predictable.
- [ ] Allow manual dark-mode overrides after generation, but keep the editing surface opinionated.
- [ ] Add reset actions for custom theme values.
    - Reset custom colors
    - Revert to preset theme
- [ ] Add validation and guardrails for unreadable or unstable color combinations.
- [ ] Add/update tests for token generation and custom theme persistence.

Done definition:
- [ ] Users can personalize the app with a few key colors while the app still produces a polished, readable light and dark theme.

---

## 4. Accessibility Controls
Parent status: `[ ] Planned`

- [ ] Add a high-contrast mode.
    - Treat this as a real token layer, not just a minor dark-mode tweak.
- [ ] Add font-size or UI scale controls in Settings.
    - Apply scaling through shared/root sizing variables rather than ad hoc per-component overrides.
    - As is best, want to make sure any measurement that makes sense to be is in rem units, so that things scale naturally based on the user's preferred font size
- [ ] Audit shared components for scaling behavior.
    - Buttons, inputs, pills, headers, cards, modal content, and table-like layouts should remain usable.
- [ ] Identify and clean up hard-coded font sizes or spacing that break accessibility scaling.
- [ ] Consider reduced-motion support if the app’s motion patterns justify it during implementation.
- [ ] Add/update tests where necessary.

Done definition:
- [ ] Users can increase readability and contrast without the app becoming visually broken or inconsistent.

---

## 5. Zoom Controls and Shortcut Support
Parent status: `[ ] Planned`

- [ ] Add app zoom in, zoom out, and reset zoom actions.
    - Cmd/Ctrl `+`
    - Cmd/Ctrl `-`
    - Cmd/Ctrl `0`
- [ ] Add the same actions to the Electron View or Settings-related menu structure.
- [ ] Preserve the current shortcut reliability model.
    - Prefer Electron/app-level handling for global reliability, with renderer support only where needed.
- [ ] Make the difference between zoom and font scaling clear in the UI.
    - Zoom affects the whole app viewport.
    - Font scaling is a user preference for readability.
- [ ] Add the shortcuts to the keyboard shortcuts modal.
- [ ] Add/update tests where necessary.

Done definition:
- [ ] Users can reliably zoom the app with standard shortcuts and understand how zoom differs from font scaling.

## 6. Settings Modal Information Architecture
Parent status: `[ ] Planned`

- [ ] Reorganize Settings sections to better support the expanded feature set.
    - Appearance
    - Accessibility
    - Glossary
    - App Data / Reset
- [ ] Ensure appearance and accessibility controls feel previewable, understandable, and reversible.
- [ ] Add concise helper copy where distinctions may be confusing.
    - Theme mode vs preset family
    - Preset theme vs custom theme
    - Font scale vs zoom
- [ ] Keep the control surface clean and opinionated.
    - Avoid an “expert mode” UI in this release.
- [ ] Add/update tests where necessary.

Done definition:
- [ ] The Settings modal can support the new appearance and accessibility controls without becoming cluttered or confusing.
