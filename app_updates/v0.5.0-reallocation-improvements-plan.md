# Reallocation Helper Improvements

> Extends the existing automated reallocation system with slider-based manual overrides, a live equilibrium progress bar, an auto-balance mode, and a smarter proportional algorithm. The goal is to give users fine-grained control over how they recover from a negative balance while keeping the automated suggestions as a useful starting point.

---

## Problem Statement

### 1. Binary check/uncheck gives no middle ground

The current review modal presents a flat checkbox list. Each proposal is either fully accepted (apply the algorithm's suggested amount) or fully rejected (skip it entirely). There is no way to say "reduce this savings goal by half instead of pausing it" or "free $50 from this item but not the full $120."

### 2. The algorithm over-concentrates changes on a single item

Within each category, candidates are sorted **largest amount first** and processed greedily. This means the first candidate absorbs the entire shortfall before the algorithm ever touches the second. A user with three savings goals at $200/paycheck each will see a proposal to pause the first goal entirely ($200 freed) even when the shortfall is $180 — one item takes the full hit while the other two are untouched.

### 3. Fixed category priority ignores item importance

Bills and deductions are always proposed before savings, which are always proposed before retirement. A user might strongly prefer to reduce a retirement contribution slightly rather than pausing several discretionary bills. The fixed ordering offers no way to express this preference.

### 4. No live sense of "enough"

The summary cards (Current Remaining, Target Remaining, Freed by Selected) are static snapshots. There is no immediate visual feedback as the user adjusts selections showing how close they are to resolving the shortfall.

### 5. No way to protect an item

Savings goals and retirement contributions have no way to opt out of reallocation suggestions permanently. Users must remember to uncheck them every time the modal opens.

---

## Solution Overview

Five focused improvements, all confined to the reallocation service and its review modal UI:

1. **Slider overrides** — Replace the binary checkbox with a per-row amount slider for reducible items (savings, retirement, custom allocations). Bills and deductions remain toggle-only (pause-only in source model).
2. **Live progress bar** — Replace the "Freed by Selected Changes" static card with a live shortfall progress bar that updates as sliders move or toggles change.
3. **Auto-balance mode** — An opt-in toggle that propagates slider changes across other adjustable items to keep total freed at the target shortfall.
4. **Proportional spreading algorithm** — Replace the greedy largest-first pass with a proportional pass that shares reductions across all items of the same type before escalating to the next category.
5. **Protected item flag** — A `reallocationProtected` field on savings contributions and retirement elections that excludes them from candidate generation entirely.

### What this plan does NOT do (intentionally)

- No changes to the `.pplan` file schema beyond the optional `reallocationProtected` field.
- No changes to `applyReallocationPlan` except to consume the user's override amounts.
- No new React context — slider/override state lives in the review modal as local state.
- No changes to the trigger logic (negative balance detection and confirm dialog remain unchanged).

## Architecture

### Current flow (unchanged)

```
createReallocationPlan(input)          → ReallocationPlan  (algorithm proposals)
  ↓ user checks/unchecks in modal
applyReallocationPlan(input, plan)     → AppliedReallocationResult
  ↓
updateBudgetData(applied)
```

### New flow (additive)

```
createReallocationPlan(input)                → ReallocationPlan  (algorithm proposals)
  ↓ user adjusts sliders or toggles in modal
buildOverriddenPlan(plan, overrideMap)        → ReallocationPlan  (user-adjusted proposals)
  ↓
applyReallocationPlan(input, overriddenPlan) → AppliedReallocationResult
  ↓
updateBudgetData(applied)
```

`buildOverriddenPlan` is a pure function that replaces `freedPerPaycheckAmount` and `proposedPerPaycheckAmount` in each proposal with the user's override, then recalculates `totalFreedPerPaycheck` and `projectedRemainingPerPaycheck`. `applyReallocationPlan` already reads `proposedPerPaycheckAmount` directly — no changes needed there.

## Step B — Auto-balance mode

**Files changed:** `ReallocationReviewModal.tsx`, `ReallocationReviewModal.css`

- [ ] Add a toggle in the modal header area: **"Auto-balance remaining items"** (off by default).
- [ ] When the toggle is on, implement propagation on each slider change or toggle flip:
  - Calculate `delta = totalFreed - shortfallPerPaycheck` after the change.
  - If `delta > 0` (over-freed): proportionally reduce other adjustable items. For each other item with freed > 0: `item.freed -= delta × (item.freed / totalOfOtherFreed)`, clamped to `[0, item.max]`.
  - If `delta < 0` (under-freed): fill the gap from remaining candidates in priority order (bills → deductions → savings → retirement), increasing each item's freed amount up to its maximum until the gap is closed.
  - Never adjust the item the user just changed.
- [ ] A "Reset to suggestions" button restores all sliders/toggles to the algorithm's original proposed values.

Done definition:
- [ ] Turning on auto-balance and moving a slider causes other sliders to adjust so the total freed stays at the shortfall target.
- [ ] Auto-balance never moves a slider the user just explicitly set.
- [ ] Reset button restores algorithm defaults.

## Step C — Proportional spreading algorithm

**Files changed:** `reallocationPlanner.ts`, `reallocationPlanner.test.ts`

- [ ] Replace the greedy loop in `createReallocationPlan` with a **type-grouped proportional pass**:
  - Group candidates by category: `[bills, benefits, customAllocations, savings, investments, retirement]`.
  - Process each group in order. Within a group, check if the combined maximum of all items covers the remaining shortfall.
    - If yes: distribute the shortfall proportionally across the group by `item.max / groupTotal`. Each item gets a partial reduction, none is fully paused unless its share rounds to its full amount.
    - If no: take the full amount from every item in the group, move the remainder to the next group.
  - After all groups are exhausted, record `fullyResolved = remainingShortfall < 0.01`.
- [ ] Switch bill/deduction candidates to **smallest-amount-first** sort (currently largest-first). Pausing smaller items is less disruptive.
- [ ] Keep savings/investment/retirement candidates at **largest-first** sort, since proportional spreading already distributes the load — the order only affects tie-breaking.
- [ ] Update `reallocationPlanner.test.ts` to reflect new proposal shapes (partial reductions instead of single-item full pauses).
- [ ] Add tests verifying proportional distribution, partially-covered group, fully-covered group, and multi-group escalation.

Done definition:
- [ ] Given three savings items at $200/paycheck each and a $180 shortfall, three proposals are generated, each reducing by $60, rather than one proposal pausing the first.
- [ ] Bills are proposed in smallest-first order.
- [ ] All new and updated tests pass.
