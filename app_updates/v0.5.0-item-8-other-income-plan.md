## Goal
Add support for income sources beyond the base salary/hourly paycheck while keeping pay calculations, taxes, summaries, exports, search, and audit history understandable and deterministic.

## Scope Decision for v0.5.0
- In scope: Fixed-amount other income entries with frequency, income type, notes, enable/disable, and taxability controls.
- In scope: Percentage-of-gross income rules for bonus-style income.
- In scope: Automated withholding presets by income type with manual override preserved.
- In scope: Clear income-impact treatment so entries can affect gross pay, taxable income only, or net pay only depending on the income source.
- In scope: Inclusion of other income impacts in annual, monthly, and per-paycheck summary views.
- In scope: History/audit coverage, search integration, save/load/migration support, and demo-data/new-plan compatibility.
- Out of scope for v0.5.0 unless explicitly approved later: Country-specific tax engines for all jurisdictions tied to other-income type.
- Out of scope for v0.5.0 unless explicitly approved later: Fully date-effective payout schedules beyond simple frequency/month-window controls.

## Product Model

### Supported Income Types
- [ ] Add built-in income types for Bonus, Commission, Rental Income, Retirement Withdrawal, Disability, Reimbursement/Allowance, Investment Income, and Other.
- [ ] Allow a custom label when `incomeType = 'other'` or when users want a payroll-source-specific name.

### Income Impact Modes
- [ ] Add a required income-impact field, such as `payTreatment: 'gross' | 'taxable' | 'net'`.
- [ ] Define `gross` treatment for payroll-like income that should increase gross pay first, then flow through taxable income and taxes.
- [ ] Define `taxable` treatment for supplemental income that should affect withholding/taxable wages but remain visually separate from base paycheck gross.
- [ ] Define `net` treatment for after-tax income that should increase take-home totals without affecting taxable income or withholding.
- [ ] Document which built-in income types default to each treatment and allow manual override when needed.

### Withholding Behavior
- [ ] Add per-entry withholding mode, such as `manual | auto | none`, with sane defaults by income type and treatment.
- [ ] Ensure `auto` mode can propose withholding deltas without overwriting the user’s existing manual tax lines.
- [ ] Ensure `none` mode is available for net-only and user-managed tax scenarios.

### Calculation Principles
- [ ] Keep calculations deterministic and explainable across mixed base pay + other income scenarios.
- [ ] Preserve the current bias toward conservative/over-estimated withholding instead of under-estimated take-home.
- [ ] Keep manual user edits authoritative whenever automated logic would otherwise conflict.

## Workstream A: Core Other-Income Model and Data Layer

### Data Model
- [ ] Add a new `OtherIncome` type with fields for `id`, `name`, `incomeType`, `amountMode`, `amount`, `percentOfGross`, `frequency`, `enabled`, `notes`, `isTaxable`, `payTreatment`, `withholdingMode`, and optional `withholdingProfileId`.
- [ ] Add optional scheduling controls for expanded scope, such as `activeMonths?: number[]`.
- [ ] Add optional guardrails for percent-based entries, such as cap/floor fields if needed.
- [ ] Add `otherIncome?: OtherIncome[]` to `BudgetData`.
- [ ] Add migration defaults so existing plans load with `otherIncome: []`.
- [ ] Update runtime validation/type guards to safely load and validate the new structure.
- [ ] Add migration and backwards-compatibility tests.

### Defaulting and Validation Rules
- [ ] Require non-negative values for fixed amounts, percent rules, and optional caps/floors.
- [ ] Restrict percent-of-gross entries to sensible ranges and document the final guardrails.
- [ ] Require explicit frequency for every other-income entry.
- [ ] Require explicit `payTreatment` so each entry’s effect on calculations is unambiguous.
- [ ] Validate that active-month schedules are unique, in-range, and compatible with the selected frequency.

## Workstream B: Calculation Engine and Income Placement Rules

### Frequency and Annualization
- [ ] Add shared utilities to convert other-income entries into annual, monthly, and per-paycheck equivalents.
- [ ] Support deterministic handling for annual, quarterly, monthly, and per-paycheck income frequencies.
- [ ] Document rounding strategy for line-level and aggregate calculations.

### Fixed-Amount Entries
- [ ] Integrate fixed-amount other income into the pay calculation pipeline.
- [ ] Ensure `gross` entries increase gross pay before pre-tax deductions and taxes.
- [ ] Ensure `taxable` entries increase taxable income and withholding calculations while remaining distinct from base paycheck gross in presentation.
- [ ] Ensure `net` entries increase take-home amounts only and do not affect taxable income or tax calculations.

### Percentage-of-Gross Rules
- [ ] Add `amountMode: 'fixed' | 'percent-of-gross'`.
- [ ] Resolve the percent base from the base paycheck gross before other-income additions unless a future basis override is explicitly approved.
- [ ] Compute percent-based entries in a deterministic order that prevents circular references.
- [ ] Add preview helpers so the UI can show computed per-paycheck/month/year amounts before save.
- [ ] Add focused tests for mixed fixed + percent income stacks.

### Tax and Withholding Interaction
- [ ] Define how taxable other income contributes to the existing tax-estimation and tax-line model.
- [ ] For `gross` and `taxable` entries, compute withholding deltas separately from base paycheck taxes so the source of withholding remains explainable.
- [ ] Keep current manual tax lines authoritative unless a dedicated auto-withholding path is enabled.
- [ ] Make sure auto-withholding suggestions add deltas predictably rather than silently mutating unrelated tax lines.
- [ ] Ensure `net` entries never inflate gross/taxable displays or withholding totals.

## Workstream C: Automated Withholding Presets

### Preset Model
- [ ] Add default withholding profiles by income type, such as supplemental bonus behavior, retirement-withdrawal withholding, or no-withholding defaults for non-payroll income.
- [ ] Add optional `withholdingProfileId` to other-income entries.
- [ ] Keep profile mapping internal/testable so the UI only consumes structured profile metadata.

### Calculation Behavior
- [ ] Add `withholdingMode: 'manual' | 'auto' | 'none'` per entry or equivalent plan-safe design.
- [ ] In `auto` mode, generate withholding suggestions from the selected/default profile.
- [ ] In `manual` mode, let the user enter or leave withholding impact fully under their control.
- [ ] In `none` mode, record the income without any automatic withholding impact.
- [ ] Ensure auto-generated withholding is visible as a distinct source in breakdowns and explain views.

### Explainability and Safety
- [ ] Add a trace/explain structure: income source, treatment mode, withholding source, taxable base, applied rate, and estimated withheld amount.
- [ ] Preserve a one-click fallback to manual control if the user dislikes or distrusts auto behavior.
- [ ] Add regression tests ensuring manual tax lines are not overwritten unexpectedly.

## Risks
- Risk: Adding `taxable-only` income can create confusing gross vs taxable reconciliations.
- Mitigation: Present base gross and supplemental taxable income separately in summaries and breakdowns.
- Risk: Auto-withholding may reduce user trust if it silently changes take-home pay.
- Mitigation: Keep manual override explicit, show previews, and expose the withholding source/rate/base.
- Risk: Percent-of-gross rules can introduce circular or order-dependent math.
- Mitigation: Lock the percent base to base paycheck gross and test deterministic ordering.
- Risk: Scope expansion may create regressions in core paycheck calculations.
- Mitigation: Add focused regression coverage for mixed income/tax scenarios before release cut.

## Dependencies
- Depends on stable budget migration and validation paths in file storage.
- Depends on calculation services that can support additional income inputs without destabilizing existing paycheck flows.
- Depends on tab/search registration updates for discoverability.
- Depends on export and audit layers being able to surface the new income impacts clearly.
