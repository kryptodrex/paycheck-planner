# V2 Plan: Date-Anchored Pay Plans (Calendar-Aware Projections)

## Goal
Make the entire pay plan calendar-aware. Every amount the user sees — base pay, deductions, taxes, net pay, other income — should reflect what actually happens in a real selected period, not a uniform average. March 2026 has 2 bi-weekly paychecks; the plan should show exactly that, not the 2.17 average. Other income scheduling and effective-dated amounts are then built on this same foundation.

## Problem Statement
Currently all non-annual display modes divide the annual amount by a fixed yearly divisor (`annual / 12` for monthly, `annual / 26` for bi-weekly, etc.). This produces a mathematically convenient average but misrepresents reality. A bi-weekly worker in a 3-paycheck month receives 50% more base pay that month than in a 2-paycheck month. Bills and deductions still come due monthly regardless of paycheck count. Cashflow planning is impossible without knowing what actually lands in a given month.

`firstPaycheckDate` already exists in `PaySettings` — the anchor is there. The gap is that nothing in the calculation or display pipeline uses it.

## Success Criteria
- User can opt into a calendar-accurate view that picks a specific month or paycheck and shows real amounts for that period.
- All plan amounts — base pay, deductions, taxes, other income, net — reflect the actual paycheck count in the selected period.
- Averaged mode is preserved exactly as-is for planning workflows that prefer uniform smoothing.
- Other income scheduled events land in the correct calendar periods.
- Architecture supports future effective-dated salary/deduction changes without a rewrite.
- Existing plans migrate safely; plans without `firstPaycheckDate` fall back to averaged mode with a prompt to configure.
- All views (Pay Breakdown, Key Metrics, Bills, Accounts) use the same period context so numbers are consistent.

## The Core Primitive: Pay Calendar Engine

`firstPaycheckDate` + `payFrequency` is sufficient to generate every actual paycheck date in a rolling window (typically current year ± 1 or configurable range). This engine becomes the single source of truth for any period-aware calculation in the app.

```
payCheckDates = generatePaycheckCalendar(firstPaycheckDate, payFrequency, windowStart, windowEnd)
paychecksInPeriod(month, year) => payCheckDates.filter(d => d is in that month)
```

Special cases to handle deterministically:
- Semi-monthly: anchor to configured day-of-month pair, clamp 29/30/31 to last day of month.
- Bi-weekly: pure rolling interval from anchor.
- Monthly: anchor to configured day, clamp to last day of month.
- Yearly / quarterly / semi-annually: direct date calculation.

All date math must be date-only (no timezone offsets), operating on ISO date strings (`YYYY-MM-DD`).

## Scope

### 1. Pay Calendar Engine (`src/utils/payCalendar.ts`)
- `generatePaycheckCalendar(firstPaycheckDate, frequency, windowStart, windowEnd): string[]`
  - Returns sorted ISO date strings for all paycheck dates in the window.
- `getPaychecksInPeriod(calendar, periodStart, periodEnd): string[]`
  - Filters calendar to a specific period (month, quarter, etc.).
- `getPaychecksInMonth(calendar, year, month): string[]`
  - Convenience wrapper used throughout the app.
- `getPeriodBoundaries(year, month): { start: string; end: string }`
  - Consistent period edge definitions to avoid off-by-one across call sites.
- Fully unit-tested, no external dependencies, pure functions only.
- Window defaulting: generate 2 full years (current year + next year) on first use.

### 2. Period Context (`src/contexts/PeriodContext.tsx` or integrated into `BudgetContext`)
- App-level state: `viewMode: 'averaged' | 'calendar'` and `selectedPeriod: { year: number; month: number }`.
- Derived: `paychecksInSelectedPeriod: string[]` (actual dates).
- `selectedPeriod` defaults to the current month.
- `viewMode` persists to user preferences (not the plan file).
- Navigation controls: previous/next period, "today" reset.
- When `viewMode === 'averaged'`, all existing calculation paths remain unchanged.

### 3. Calendar-Mode Calculation Layer
- New exported function `calculateCalendarPeriodBreakdown(breakdown, paychecksInPeriod)`:
  - Takes the per-paycheck breakdown and multiplies all amounts by `paychecksInPeriod.length`.
  - Returns the same `PayBreakdownSummary` shape — no new types required in the display layer.
- Replaces the `calculateDisplayPayBreakdown` path when `viewMode === 'calendar'`.
- Other income (averaged entries): `perPaycheck × paychecksInPeriod.length`.
- Other income (payout-timed entries): event-based total for events that fall in `selectedPeriod` (see §5).
- Deductions: cadence-aware amounts per §4 — `monthly-fixed` deductions are not multiplied by paycheck count; `per-paycheck` deductions are.
- Bills: monthly-fixed amounts stay monthly-fixed regardless of paycheck count.

### 4. Deduction Cadence (Benefit Deduction Holidays)

In bi-weekly payroll, benefit deductions (health, dental, vision, FSA, HSA, flat-dollar 401k, life insurance) are commonly priced as a monthly cost split across **2 paychecks per month — 24 deductions per year, not 26**. The 3rd paycheck in a 3-paycheck month is a "deduction holiday" for those line items; the worker's net pay is higher that paycheck.

This is not a quirk — it is the standard benefit-billing practice at most employers that run bi-weekly payroll. Percentage-based contributions (e.g., 401k % of gross, FICA) still apply every paycheck.

**Model:**
- Add `deductionCadence: 'every-paycheck' | 'monthly-fixed'` to the `Deduction` type.
  - `every-paycheck`: deducted on all `paychecksInPeriod` (e.g., federal tax, percentage 401k). Default for all existing deductions.
  - `monthly-fixed`: deducted on the **first 2 paychecks of the month only**, regardless of how many paychecks the month contains. Annual total = `monthlyAmount × 12`.
- In calendar mode, `calculateCalendarPeriodBreakdown` applies `monthly-fixed` deductions at their monthly flat amount regardless of paycheck count (2-paycheck month and 3-paycheck month show the same deduction total).
- In averaged mode, `monthly-fixed` deductions continue to use `annualAmount / displayModeDivisor` — no behavior change.
- `fileStorage.ts` normalizes `deductionCadence` and defaults to `'every-paycheck'` for all existing deductions on file load.

**Other cadence patterns to consider (document now, implement with §4):**
- Some employers skip deductions on the **last** paycheck of the month instead of the 3rd. UI should allow the user to configure which paycheck is skipped: `skipPaycheck: 'third' | 'last'`.
- Semi-monthly workers always have exactly 2 paychecks and never experience a deduction holiday — `monthly-fixed` and `every-paycheck` produce identical results for them.
- Weekly payroll (52/year): similar dynamic — benefits may be billed as monthly deducted on first 4 paychecks, with a 5th-paycheck holiday in 5-paycheck weeks.

### 5. Other Income Event Scheduling (builds on §1)
- Per-entry scheduling: define payout anchor date(s) for timed entries.
  - Yearly: month + day.
  - Quarterly: seed month + day (auto-computes 4 dates).
  - Monthly: day of month (clamp to end of month if needed).
  - Per-paycheck cadence: tied to a specific paycheck number (e.g., "2nd paycheck of month") using the calendar from §1.
- `generateOtherIncomeEvents(entry, calendar, windowStart, windowEnd): { date: string; amount: number }[]`
  - Deterministic, pure, fully testable.
- In calendar mode, `calculateCalendarPeriodBreakdown` aggregates events within the period boundaries.
- In averaged mode, existing `calculateOtherIncomePerPaycheckAmount` behavior is unchanged.

### 6. UI: Period Selector and Mode Toggle
- Period selector component: month/year picker with prev/next navigation.
  - Displays `"March 2026 · 2 paychecks"` to make the count explicit.
  - Shows `"Average"` label and no date when in averaged mode.
- Mode toggle: `"Averaged"` vs `"Calendar"` — placed near the display mode switcher.
  - Only enabled when `firstPaycheckDate` is configured.
  - If not configured, button is disabled with tooltip: "Set your first paycheck date in Pay Settings to enable calendar view."
- Pay Breakdown header updates to reflect active period and paycheck count.
- Key Metrics shows period-accurate gross/net/savings.
- Bills Manager shows impact toggle: which bills are due this month vs. monthly average.

### 7. Pay Settings: First Paycheck Date
- `firstPaycheckDate` already exists in the schema — validate it is captured in the setup wizard and Pay Settings form.
- Add inline helper text explaining its role: "Used to calculate your exact paycheck dates for each month."
- For semi-monthly plans: validate that both `semiMonthlyFirstDay` and `semiMonthlySecondDay` are configured, since period accuracy depends on them.
- Surface a non-blocking banner in Pay Breakdown when `firstPaycheckDate` is missing and `viewMode === 'calendar'` is attempted.

### 8. Persistence and Migration
- `viewMode` preference and `selectedPeriod` are user-level (not plan-level) — do not persist to `.pplan` files.
- No migration needed for existing plan files; `firstPaycheckDate` field already exists.
- `deductionCadence` defaults to `'every-paycheck'` for all existing deductions — no data loss, no behavior change in averaged mode.
- Scheduling fields added to `OtherIncome` in §5 are new optional fields with safe defaults (null = no schedule = averaged behavior).
- `fileStorage.ts` normalization: validate and default `deductionCadence` and new schedule fields.

### 9. Audit History, Search, Export
- History snapshots: if a snapshot was taken in calendar mode, record the `selectedPeriod` and `paychecksInPeriod` count in the snapshot metadata.
- History cards: render period context ("March 2026 — 2 paychecks") alongside amounts when metadata is present.
- PDF export: include selected period and mode in the report header.
- Search: other income subtitle shows next scheduled payout date when applicable.

### 10. Foundation for Effective-Dated Amounts (documented, not yet implemented)
The calendar engine built in §1 makes effective-dated changes straightforward to add later:
- A salary change effective 2026-07-01 simply means: use `baseGross_v1` for paychecks before that date, `baseGross_v2` for paychecks on or after it.
- `calculateCalendarPeriodBreakdown` would partition the paycheck list by effective date ranges, compute each segment, and sum.
- Deduction start/end dates follow the same pattern.
- No architectural change is required in V2 to support this — the calendar already provides the paycheck date list.

## Test Strategy
- **Unit (pay calendar):** All frequency types, anchor date edge cases (month-end, year-end, leap day, day clamping for semi-monthly), correct paycheck counts per month across a full year. Explicitly test known 3-paycheck months for a given anchor.
- **Unit (calculations):** Averaged vs. calendar mode parity for annual total; 3-paycheck month gross = 3× per-paycheck; `monthly-fixed` deduction in 3-paycheck month equals `monthly-fixed` deduction in 2-paycheck month (deduction holiday); `every-paycheck` deduction in 3-paycheck month = 3× per-paycheck deduction; mixed averaged + timed other income in same period.
- **Service tests:** File normalization for new schedule fields and `deductionCadence`; missing `firstPaycheckDate` graceful fallback; existing deduction records default to `'every-paycheck'`.
- **Component tests:** Period selector navigation; paycheck count display; Pay Breakdown amounts update on period change; `firstPaycheckDate` missing banner; deduction cadence control in Benefits Manager; OtherIncome schedule controls and next-event preview.
- **Snapshot regression:** History cards rendering period metadata; deduction holiday shown in Pay Breakdown for 3-paycheck period.

## Risks
- Date-only math drift if any calculation accidentally involves `Date()` constructors with local timezone offsets.
- Bi-weekly edge: 3-paycheck months occur roughly 2× per year — tests must exercise both 2-paycheck and 3-paycheck months explicitly.
- Deduction cadence misconfiguration: if a user marks a percentage-based deduction (e.g., 401k % of gross) as `monthly-fixed`, the annual math will be wrong — validation must warn or prevent this.
- Not all employers skip the 3rd paycheck deduction — some run all deductions every paycheck and simply use a 26-period benefit cost. The cadence setting must be user-controlled, not auto-detected.
- Semi-monthly always produces exactly 2 paychecks per month, but day-clamping (e.g., Feb 30 → Feb 28) must be consistent.
- User confusion if calendar mode is on but `firstPaycheckDate` is not set — hard to diagnose incorrect amounts.
- Bills with irregular cadence (quarterly insurance, annual fees) need correct period bucketing.

## Mitigations
- All date math operates on ISO date string arithmetic with no `Date` timezone involvement.
- Period arithmetic helper functions are independently unit-tested with fixed known-good cases before integration.
- `deductionCadence` defaults to `'every-paycheck'` for all existing deductions — averaged mode behavior is identical to current behavior for all existing data.
- Warn in Benefits Manager if user sets `monthly-fixed` on a deduction that uses a percentage amount mode.
- `firstPaycheckDate` absence blocks calendar mode at the UI level with a clear explanation.
- Memoize `generatePaycheckCalendar` output keyed on `(firstPaycheckDate, frequency, windowStart, windowEnd)`.
- Averaged mode is the default; calendar mode is an opt-in that users deliberately enable.

## Out of Scope (for V2)
- Effective-dated salary or deduction changes (architecture supports it, implementation deferred to V3).
- Business-day adjustment (paycheck lands on Friday if it falls on a weekend) — document as a known gap.
- Auto-detection of which employer uses deduction holidays vs. 26-period benefit billing — user-configured only.
- Tax withholding automation per timed other-income event.
- External payroll calendar imports.
- Locale-specific holiday or business-day rules.
