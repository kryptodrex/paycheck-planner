## 1. Centralize Tab Identifiers
Parent status: `[x] Done`

**Problem:** Tab IDs (`'metrics'`, `'breakdown'`, `'bills'`, `'loans'`, `'taxes'`, `'savings'`, `'other-income'`) are scattered as raw string literals across ~15 files. Renaming or adding a tab requires hunting through search modules, PlanDashboard routing, PlanTabs, and test fixtures.

**Files affected:**
- [src/utils/tabManagement.ts](../src/utils/tabManagement.ts) — `TabId` type union defined here, but IDs are hardcoded strings
- [src/components/PlanDashboard/PlanDashboard.tsx](../src/components/PlanDashboard/PlanDashboard.tsx) — 20+ tab ID string comparisons in `activeTab` routing, `openTabFromLink`, keyboard handlers
- [src/utils/searchModules/](../src/utils/searchModules/) — Each search module hardcodes its tab ID (e.g. `tabId: 'bills'`) with no shared constant

**Tasks:**
- [x] Create a `TAB_IDS` constant object in `src/constants/tabIds.ts` (e.g. `TAB_IDS.bills`, `TAB_IDS.metrics`) and derive the `TabId` type from it.
- [x] Replace all raw tab ID string literals across PlanDashboard, search modules, and tests with `TAB_IDS.*` references.
- [x] Verify the `TabId` type is derived from the constant (`type TabId = typeof TAB_IDS[keyof typeof TAB_IDS]`) so adding a new tab is a single-location change.
- [x] Add/update tests where necessary.

Done definition:
- [x] No raw tab ID string literal exists outside `src/constants/tabIds.ts` and test assertions.
- [x] Adding a new tab ID requires only one new entry in the constants file.

## 2. Centralize Frequency Constants
Parent status: `[x] Done`

**Problem:** Pay frequency, bill frequency, savings frequency, and view mode strings are used as discriminators in 30+ switch/if-else statements across 12+ files. The types exist in `src/types/frequencies.ts` and `src/types/viewMode.ts`, but every function that branches on them uses raw string literals like `'bi-weekly'` or `'monthly'`.

**Files affected:**
- [src/utils/payPeriod.ts](../src/utils/payPeriod.ts) — 5+ switch statements on frequency/view mode strings (`getPaychecksPerYear`, `getDisplayModeLabel`, `getDisplayModeOccurrencesPerYear`, `getPayFrequencyViewMode`, `formatPayFrequencyLabel`)
- [src/utils/payCalendar.ts](../src/utils/payCalendar.ts) — 5 separate case statements on pay frequency
- [src/utils/frequency.ts](../src/utils/frequency.ts) — `PAY_FREQUENCY_OCCURRENCES`, `BILL_FREQUENCY_OCCURRENCES`, `SAVINGS_FREQUENCY_OCCURRENCES` maps with hardcoded keys
- [src/utils/billFrequency.ts](../src/utils/billFrequency.ts) — `convertBillToMonthly`, `formatBillFrequency` switch on frequency strings
- Component files — UI option arrays for frequency dropdowns duplicate the same strings

**Tasks:**
- [x] Create `src/constants/frequencies.ts` exporting constant objects for each frequency family (e.g. `PAY_FREQUENCIES.biWeekly`, `BILL_FREQUENCIES.semiAnnual`) and derive the union types from them.
- [x] Refactor switch statements in `payPeriod.ts`, `payCalendar.ts`, `frequency.ts`, and `billFrequency.ts` to use the constants instead of raw strings.
- [x] Consolidate frequency dropdown option arrays (currently defined inline in OtherIncomeManager, LoansManager, etc.) into shared constants that components import.
- [x] Do the same for `ViewMode` strings — create a `VIEW_MODES` constant object and derive the type.
- [x] Add/update tests where necessary.

Done definition:
- [x] No raw frequency or view mode string literal used as a discriminator outside the constants file.
- [x] Dropdown option arrays for frequencies are imported from a shared constant, not defined inline per component.

## 6. Remove Component Import from BudgetContext
Parent status: `[x] Done`

**Problem:** [BudgetContext.tsx](../src/contexts/BudgetContext.tsx) imports `ErrorDialog` from `src/components/_shared`. This is a dependency-inversion violation — the data/state layer should not depend on presentation components. It makes BudgetContext harder to test in isolation and creates a potential circular dependency risk.

**Tasks:**
- [x] Refactor BudgetContext to accept an error display callback (or emit an error event) instead of directly rendering `ErrorDialog`.
- [x] Move `ErrorDialog` rendering to a higher-level component (e.g. a provider wrapper or PlanDashboard) that subscribes to BudgetContext error state.
- [x] Add/update tests where necessary.

Done definition:

## 8. Improve Logic Gate Robustness
Parent status: `[x] Done`

**Problem:** Many switch statements across the codebase use string literal cases with no `default` exhaustiveness check. If a new value is added to a union type, the compiler won't flag the missing case — it silently falls through.

**Key locations:**
- [src/utils/payPeriod.ts](../src/utils/payPeriod.ts) — 5+ switch functions (`getPaychecksPerYear`, `getDisplayModeLabel`, etc.)
- [src/utils/payCalendar.ts](../src/utils/payCalendar.ts) — Switch on `PayFrequency`
- [src/utils/billFrequency.ts](../src/utils/billFrequency.ts) — `convertBillToMonthly`, `formatBillFrequency`

**Tasks:**
- [x] Add exhaustive `default: never` checks (or use a helper like `assertNever()`) to all switch statements that branch on union types.
    - This causes a compile error if a new union member is added but not handled.
- [x] Where feasible, replace switch statements with constant lookup maps (e.g. `Record<PayFrequency, number>`) so missing keys are caught by the type system at definition time rather than at each call site.
- [x] Add/update tests where necessary.

Done definition:
- [x] All switch statements on discriminated union types have exhaustive handling or `assertNever()` default.
- [x] Adding a new member to a frequency, view mode, or tab type causes a compile error at every unhandled site.
