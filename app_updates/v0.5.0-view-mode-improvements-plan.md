# View Mode Improvements: Calendar-Accurate Periods

> Pared-down replacement for the scrapped [Item 5 (Date-Anchored Plans)](v0.5.0-item-5-date-anchored-plans.md). That plan tried to build a full calendar engine with deduction cadences, other-income event scheduling, period contexts, and more — it was over-engineered and confusing. This plan focuses on the two concrete problems that actually matter.

## Problem Statement

### 1. Monthly and quarterly views are averages, not actuals

The current display pipeline converts amounts like this:

```
per-paycheck → × paychecksPerYear → annual → ÷ displayOccurrences → display
```

Monthly view = `annual / 12`. Quarterly = `annual / 4`. These are uniform averages.

For a bi-weekly worker (26 paychecks/year), some months contain 2 paychecks and others contain 3. The monthly view always shows `annual / 12`, which doesn't match either scenario. A 3-paycheck month means ~50% more take-home than a 2-paycheck month — that difference matters for real budgeting.

Same issue for weekly workers: some months have 4 paychecks, others have 5.

Semi-monthly and monthly pay frequencies are unaffected — they always produce a fixed number of paychecks per month.

### 2. Per-paycheck account allocation is missing

Bills are stored as monthly amounts and displayed via `monthlyToDisplayAmount()`. If I have $1,200/month in bills from Account A and I'm paid bi-weekly, how much should I move into that account every paycheck? The answer is `$1,200 × 12 / 26 = $553.85` — a single stable amount every paycheck, regardless of whether the month has 2 or 3 paychecks. Over 26 paychecks that equals exactly $14,400 ($1,200 × 12). The app currently doesn't surface this — you have to do the math yourself.

## Solution Overview

Three focused additions, no changes to the existing data model or file format:

1. **Pay Calendar utility** — pure functions that generate actual paycheck dates from `firstPaycheckDate` + `payFrequency` (both already exist in `PaySettings`).
2. **Calendar-accurate display mode** — opt-in toggle that makes the Monthly and Quarterly views show amounts for a specific real period (e.g., "March 2026 — 2 paychecks") instead of the annual average.
3. **Stable per-paycheck account allocation** — in the Bills Manager, show the fixed per-paycheck amount to send to each account that covers annual bills evenly, plus the recommended starting buffer.

### What this plan does NOT do (intentionally)

- No `deductionCadence` field on `Deduction`. Deduction holidays are an employer payroll detail; the user can account for them manually. If demand exists, this can be layered on later with zero architectural change.
- No other-income event scheduling. The existing `frequency` + `activeMonths` fields on `OtherIncome` are sufficient. Calendar-accurate mode simply uses `perPaycheck × paychecksInPeriod` for other income.
- No new period context. The calendar-accurate state lives in `PlanDashboard` alongside the existing `displayMode` state — same pattern, no new React context.
- No changes to the `.pplan` file schema. `firstPaycheckDate` already exists. The calendar toggle is a UI preference, not plan data.

## Architecture: How It Fits

### Current calculation pipeline (unchanged in averaged mode)

```
calculatePaycheckBreakdown(budgetData)        → PaycheckBreakdown  (per-paycheck)
calculateAnnualizedPayBreakdown(breakdown, N)  → PayBreakdownSummary (annual)
calculateDisplayPayBreakdown(annual, mode, N)  → PayBreakdownSummary (display)
```

Where `N` = `getPaychecksPerYear(payFrequency)` — a fixed number (52, 26, 24, 12, etc.).

### New calendar-accurate path (additive)

```
calculatePaycheckBreakdown(budgetData)              → PaycheckBreakdown  (per-paycheck)
calculateCalendarPeriodBreakdown(breakdown, count)   → PayBreakdownSummary (period total)
```

Where `count` = `getPaychecksInMonth(firstPaycheckDate, payFrequency, year, month)` — the actual number of paychecks in the selected month or quarter.

`calculateCalendarPeriodBreakdown` multiplies every field in the per-paycheck breakdown by `count`. Same output shape as `PayBreakdownSummary` — the display layer doesn't know the difference.

### Bills in calendar-accurate mode

Bills are monthly. A monthly bill of $500 is still $500 in calendar-accurate monthly view — no change needed. The value comes from the **income side**: seeing that your real take-home this month is `2 × netPay` or `3 × netPay`, so the gap between income and bills is accurate.

### Stable per-paycheck account allocation

The allocation uses a **stable amount every paycheck** rather than varying month-to-month:

```
stableAllocation = accountMonthlyTotal × 12 / paychecksPerYear
```

For bi-weekly (26 paychecks): `$1,200/mo × 12 / 26 = $553.85/paycheck`.

- **2-paycheck months:** `$553.85 × 2 = $1,107.69` deposited vs `$1,200` withdrawn → account draws down `$92.31`
- **3-paycheck months:** `$553.85 × 3 = $1,661.54` deposited vs `$1,200` withdrawn → account builds `$461.54` buffer

Over any 12-month period these balance out to zero. But the user needs a **starting buffer** to cover the shortfall in 2-paycheck months before a 3-paycheck month replenishes it. The recommended buffer is displayed per account:

```
recommendedBuffer = monthlyTotal - (stableAllocation × minPaychecksInAnyMonth)
```

For bi-weekly with `min = 2`: `$1,200 - ($553.85 × 2) = $92.31`. That's the most the account will ever draw below the monthly bill amount. One month's buffer (`$1,200`) is a safe conservative recommendation; the precise minimum (`$92.31`) is the tight floor.

This allocation **does not require calendar-accurate mode** — it works with just `paychecksPerYear`, which is always known. Calendar-accurate mode adds the context of *which* months are 2-paycheck vs 3-paycheck, but the stable allocation amount is the same regardless.

## Phase 1: Pay Calendar Utility
Parent status: `[x] Done`

- [x] Create `src/utils/payCalendar.ts` with the following pure functions:
    - `generatePaycheckDates(firstPaycheckDate, frequency, rangeStart, rangeEnd): string[]` — returns sorted ISO `YYYY-MM-DD` strings for all paycheck dates in range. Bi-weekly: 14-day intervals. Weekly: 7-day intervals. Semi-monthly: uses `semiMonthlyFirstDay`/`semiMonthlySecondDay`, clamps to end of month. Monthly: day-of-month from anchor, clamps to end of month. No `new Date()` timezone drift — all arithmetic on ISO strings.
    - `getPaychecksInMonth(firstPaycheckDate, frequency, year, month, semiMonthlyDays?): number` — count of paychecks falling in the given month; wrapper around `generatePaycheckDates`.
    - `getPaychecksInQuarter(firstPaycheckDate, frequency, year, quarter, semiMonthlyDays?): number` — same, filtered to 3-month quarter boundaries.
    - `getMinPaychecksInMonth(firstPaycheckDate, frequency, year, semiMonthlyDays?): number` — scans all 12 months of `year` and returns the minimum count; used by account allocation buffer calculation.

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Timezone drift in date math | All date logic uses ISO string arithmetic, no `Date()` constructors with local time. |
| User confusion when numbers change between months | Header clearly shows period + paycheck count. Toggle makes it obvious which mode is active. |
| `firstPaycheckDate` not set | Toggle disabled with tooltip. Averaged mode works exactly as before. Allocation falls back to conservative buffer. |
| Rounding: 12 calendar months don't sum to annual exactly | Expected with cent rounding. Not a bug — document in FAQ if needed. |
| Bi-weekly 27-paycheck years (rare) | Calendar utility handles naturally — generates dates, counts what falls in the range. No special case needed. |
