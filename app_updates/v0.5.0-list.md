- [x] No external HTTP client dependencies required (uses native fetch)
- [x] Environment variable properly configured in .env and Vite build

## 2. Reduce Rounding Errors in Currency Conversion
Parent status: `[x] Done`

### Phase 1: Inverse Rate Display (COMPLETE)
- [x] Auto-calculate and display inverse exchange rate as user types
  - When user enters `1 USD = 0.92 EUR`, automatically shows `Inverse: 1 EUR = 1.0870 USD`
  - Eliminates need for manual calculation in helper text
  - Updates in real-time with 8 decimal place precision stripped of trailing zeros
- [x] Integrated into Pay Details → Currency exchange rate input
  - Shows below the input field with last updated timestamp
  - Automatically clears when exchangeRate is cleared or currencies match

### Phase 2: Round-Trip Precision Analysis (COMPLETE)
- [x] Test round-trip conversion accuracy: USD → JPY → USD on $65,000 salary
  - **Finding**: Using precise inverse rate calculations (1 ÷ original_rate), round-trips are lossless to the penny
  - **Finding**: Manual inverse rounding (e.g., entering 1.177 instead of 1.17706...) causes $3+ loss on $65K salary
  - **Finding**: Live rates with 8-decimal precision (e.g., 0.92 rate) eliminate manual rounding errors
- [x] Document precision losses with manual inverse calculations
  - Added 5 comprehensive round-trip tests: EUR, JPY, GBP, multi-field, and precision degradation scenarios
  - Tests verify live rates outperform manual entry by eliminating rounding errors
- [x] Create `calculateInverseRate()` utility function in currency utils
  - Provides precise inverse rate with configurable decimal precision
  - Strips trailing zeros for clean display
  - Can be used for validation warnings or suggestions in future phases

### Phase 2 Findings & Architecture Notes:
**Round-trip accuracy without original-currency metadata:**
- Current 2-decimal rounding in `convertBudgetAmounts()` is acceptable for typical conversions
- Precise inverse calculation (1 ÷ rate) maintains accuracy to the penny automatically
- Manual inverse rounding introduces $3-$30+ losses depending on precision level
- **Recommendation**: Auto-calculate inverse as we do now; no schema change needed for Phase 2

**Why original-currency metadata storage deferred to Phase 3:**
- Live rates eliminate the primary use case (precision recovery from manual errors)
- Schema migration for metadata storage is non-trivial (versioning, backward compatibility)
- Cost/benefit favors better UX (auto-inverse display) over schema complexity

### Done definition (Phases 1 & 2):
- [x] Inverse rate is calculated and displayed to the user in the Pay Details exchange rate input
- [x] USD → JPY → USD round-trip on $65,000 salary loses $0 using precise inverse, $0-±$3 with floating-point rounding
- [x] Round-trip test suite confirms live rates + precise inverse formula prevent user-facing precision loss

## 3. Taxation API Integration or Taxation Calculation Improvements
Parent status: `[x] Done`

- [x] Improve built-in tax calculations to reduce manual effort (no API required for baseline).
  - Added `taxEstimationService` with smarter U.S. estimates:
    - Progressive federal bracket model (single-filer baseline + standard deduction)
    - Social Security wage-base capping behavior
    - Medicare surtax behavior for high income
    - Better state-rate heuristic by income band
- [x] Integrate one-click estimate action in Pay Details → Tax Lines editor.
  - New `Auto-estimate rates` action updates existing tax lines by label and keeps fields editable.
- [x] Upgrade Setup Wizard tax starter estimates to use the same smarter estimator.
- [x] Add tests for tax estimation service behavior (USD and non-USD paths, wage-base/surtax behavior).
- [x] Research external taxation APIs (US-focused, with practical fallback strategy).
  - **API Ninjas (Income Tax + Income Tax Calculator)**: easiest hosted API path; supports US/CA, filing status, deductions/credits, and premium FICA/state fields.
  - **OpenFisca**: strongest transparency/extensibility option, but requires self-hosting and rules maintenance.
  - **IRS source data**: official bracket source for annual updates; not a direct calculator API.
- [x] Evaluate feasibility, data quality, and privacy implications.
  - **Decision:** keep local estimator as default and add API as an optional enhancement only.
  - **Privacy stance:** send only derived income/tax inputs when API mode is enabled; never send account/bill/category data.
  - **Reliability stance:** always fall back to local estimator when API unavailable.
- [x] Add IRS-backed tax rule data to local app data storage (`src/data`) to avoid paid API lock-in.
  - Added local US tax rule dataset with source metadata and tax-year versioning.
  - Estimator now reads rule constants from local data files instead of hardcoded values.
  - Users can still manually edit/add tax lines in Tax Lines Editor when their jurisdiction differs.
- [x] Design optional API-assisted mode: API prefill + user-editable overrides + local fallback cache.
  - Provider order: `API Ninjas -> local estimator fallback`.
  - Cache: last successful API suggestion by `(currency, filing status, region, annual income band, tax year)` for offline reuse.
  - UX: keep `Auto-estimate rates` as one-click action; label source as `Estimated (API)` or `Estimated (Local)`.
- [x] Add/update integration tests for shipped local-estimator mode (tax estimator + paycheck calculations + demo data generation).

### Local data policy:
- Ship with local IRS-backed U.S. rules under `src/data`.
- Review/update annually for IRS tax-year changes.
- Keep UI manual edit controls as the final override layer for user-specific needs.

Done definition:
- [x] Users can auto-populate tax lines with a single action from Pay Details.
- [x] Manual editing of tax lines is preserved and not overwritten when using local auto-estimation.
- [x] Tax estimation works offline (local estimator + local rule data; no network required).
