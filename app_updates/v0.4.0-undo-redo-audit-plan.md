## Implementation Sizing

- Undo/Redo engine (app-level): Medium-High (about 1.5 to 3 dev days with tests)
- Audit history capture and viewer: High (about 2 to 4 dev days incremental)

## 1. App-Level Undo/Redo Foundation
Parent status: `[ ] Planned`

- [ ] Add history stacks to the budget state layer.
    - Add undo and redo stacks in BudgetContext.
    - Add a max history depth guard to control memory growth.
- [ ] Introduce one shared mutation helper for tracked updates.
    - Ensure user-initiated changes push prior snapshot to undo stack.
    - Ensure new edits clear redo stack.
- [ ] Add explicit support for non-tracked updates.
    - System updates (load, relink metadata, session sync, etc.) should be excluded from history by default.
- [ ] Add context API surface for `undo`, `redo`, `canUndo`, `canRedo`.
- [ ] Add/update tests for stack behavior and state restoration.

Done definition:
- [ ] A user can make edits, undo them, and redo them with deterministic data restoration across core plan entities.

## 3. Mutation Coverage and History Grouping
Parent status: `[ ] Planned`

- [ ] Audit all mutation entry points and route them through tracked updates.
    - Pay settings, deductions, taxes, accounts, bills, loans, savings, retirement, tab/settings changes that are user-editable.
- [ ] Define operation grouping rules.
    - Treat multi-field modal save as one undo step, not many.
    - Treat scripted bulk updates (reallocation apply) as one undo step.
- [ ] Define undo boundaries for setup and plan lifecycle operations.
    - New plan, open plan, close plan, copy plan should reset or initialize history as appropriate.
- [ ] Add/update tests for grouped operations and boundary behaviors.

Done definition:
- [ ] Undo/redo steps feel intuitive, granular enough for user control, and do not explode into noisy per-keystroke history entries.

---

## 4. Audit History Data Model (Object-Level)
Parent status: `[ ] Planned`

- [ ] Define an audit log schema for object-level snapshots.
    - Include timestamp, entity type, entity id, object snapshot, source action, and optional note.
    - Examples of object types we can start with: gross to net breakdown, after tax allocations, bill, deduction, savings contribution, retirement election, individual allocation item, loan, certain settings, etc.
    - Any object that is a large piece, that can be easily viewed all together.
- [ ] Capture audit entries from the same tracked mutation helper used by undo/redo.
    - Avoid duplicating logging logic across features.
- [ ] Decide where audit history is stored.
    - Preferred: in plan file metadata section (portable with plan).
    - Optional: app-level sidecar cache for very large history.
- [ ] Add safeguards to avoid logging sensitive material unnecessarily.
    - Keep encryption keys and secrets excluded.
- [ ] Add/update tests for audit-entry creation and schema stability.

Done definition:
- [ ] Object state changes for bills, deductions, savings, retirement, and allocations are captured as inspectable snapshot history.

## 5. Audit History Viewer UX
Parent status: `[ ] Planned`

- [ ] Add per-object History entry points in item cards and edit surfaces.
    - Add a History button on SectionItemCard objects (bill, deduction, savings, retirement, loan).
    - Figure out a location that makes sense for other objects like gross to net breakdown, certain settings, allocation items, etc.
- [ ] Support filtering and search.
    - Initial scope: object-local timeline with optional date filtering.
- [ ] Render object timeline using familiar card layouts.
    - Reuse existing item component layout so users can visually compare states quickly.
    - Show newest entries first, older entries below in scrollable order.
- [ ] Show date/time metadata on every history entry.
    - Include updated-at timestamp in local user-readable format.
- [ ] Allow per-object audit-entry deletion.
    - User can remove one or more history entries from that object timeline.
    - Add confirmation UX before delete and ensure deletion itself is auditable if desired.
- [ ] Add/update tests for viewer rendering and filter behavior.

Done definition:
- [ ] Users can open an object's history, scroll through its past states with timestamps, and optionally remove entries they no longer want to retain.
